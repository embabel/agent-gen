# Agent-API Integration Design

## Overview

The meta-agent framework uses a hybrid approach combining Large Language Model (LLM) intelligence with the proven Embabel Agent-API structures. This design ensures type-safe, annotation-driven agent generation while leveraging LLM capabilities for intelligent content creation.

## Core Architecture

### Agent Generation Strategy

```
UserInput → LLM Analysis → Agent-API Structures → Generated Agent Code
```

The meta-agent analyzes user requirements using LLM capabilities, then generates concrete instances of:
- `com.embabel.agent.core.Agent` - The main agent structure
- `com.embabel.agent.core.Action` - Individual action implementations  
- `com.embabel.agent.core.Goal` - Achievement targets with GOAP integration
- `com.embabel.agent.core.Condition` - Explicit condition predicates

### Key Integration Points

#### 1. Agent Structure (`Agent.kt`)
```kotlin
/**
 * Core agent structure generated by meta-agent framework.
 * 
 * @param name Generated from user intent analysis
 * @param provider Set to "meta-agent-generated" 
 * @param version Semantic version for generated agent
 * @param description LLM-generated comprehensive description

data class Agent(
    override val name: String,
    override val provider: String, 
    override val version: Semver = Semver(),
    override val description: String,
    override val conditions: Set<Condition> = emptySet(),
    override val actions: List<Action>,
    override val goals: Set<Goal>,
    val stuckHandler: StuckHandler? = null,
    override val schemaTypes: Collection<SchemaType> = inferDataTypes(...)
)
```

**LLM Generation Strategy:**
- Name: Generated from user intent analysis
- Provider: Set to "meta-agent-generated"
- Description: LLM-generated comprehensive description
- Actions: Created using Action generation process
- Goals: Created using Goal generation process
- Conditions: Generated explicit conditions for domain logic

#### 2. Action Generation (`Action.kt`)
Actions implement the `Action` interface with these key properties:
- `outputs: Set<IoBinding>` - Type-safe output bindings
- `cost: ZeroToOne` - Planning cost estimation
- `canRerun: Boolean` - Rerunability flag
- `qos: ActionQos` - Quality of service requirements

**LLM Generation Process:**
1. Analyze user requirements for action capabilities
2. Generate method signatures with proper IoBinding types
3. Create preconditions and effects for GOAP planning
4. Generate implementation code with proper annotations

#### 4. Condition Generation (`Condition.kt`)

## GOAP Conditions: Explicit vs Implicit

### Explicit Conditions (`Condition` interface)
Named, reusable predicates that implement the `Condition` interface:

```kotlin
/**
 * Generated explicit condition for domain-specific logic.
 * 
 * @param name Well-known condition identifier
 * @param cost Evaluation expense (0=cheap, 1=expensive)
 * @param evaluator Runtime evaluation logic
 */
interface Condition : Operation, HasInfoString {
    /**
     * Cost of evaluating the condition. 0 is cheap, 1 is expensive.
     * Helps in planning optimization.
     */
    val cost: ZeroToOne

    /**
     * Evaluate the condition in the context of the process.
     * May be expensive, so cost is provided for planning optimization.
     * 
     * @param context Operation context with available data
     * @return ConditionDetermination (TRUE, FALSE, UNKNOWN)
     */
    fun evaluate(context: OperationContext): ConditionDetermination
}
```

**Features:**
- Logical operators: `not()`, `or()`, `and()` for complex conditions
- Short-circuit evaluation for performance optimization
- Three-state logic: TRUE, FALSE, UNKNOWN

### Implicit Conditions (Type-based Chaining)
GOAP automatically creates implicit conditions based on type relationships:
- Action returning `Type A` → creates implicit condition "A is available"
- Action requiring `Type A` input → precondition "A must be available"
- **Automatic chaining**: `actionProducingA() → actionConsumingA(a: A)`

**Example from embabel-agent-api:**
```kotlin
/**
 * Extracts person information from user input.
 * Implicit condition: Creates "Person is available" when successful.
 * 
 * @param userInput Raw user text input
 * @return Person object if extraction successful, null otherwise
 */
@Action
fun extractPerson(userInput: UserInput): Person?

/**
 * Collects astrological details for a person.
 * Implicit precondition: Requires "Person is available" from previous action.
 * Automatic chaining occurs due to type alignment.
 * 
 * @param person The person for whom to collect star sign information
 * @return Starry object containing astrological details
 */
@Action(cost = 100.0) // High cost discourages unless necessary
fun makeStarry(person: Person): Starry
```

**Type Chaining Benefits:**
- Zero explicit condition definitions needed
- Automatic action sequence planning
- Type-safe parameter passing
- IDE support for refactoring and navigation

### LLM Generation Strategy for Conditions

#### Explicit Conditions Generation:
1. **Domain Analysis**: LLM identifies business rules and validation requirements
2. **Cost Assignment**: Evaluates complexity to assign appropriate cost values
3. **Documentation**: Generates comprehensive evaluation criteria documentation
4. **Logic Implementation**: Creates evaluation methods with proper error handling

#### Implicit Conditions (Type Chaining):
1. **Type Analysis**: LLM examines required capabilities to design data flow
2. **Signature Generation**: Creates strongly-typed method signatures for chaining
3. **IoBinding Integration**: Ensures proper type safety with Embabel framework
4. **Chain Optimization**: Designs action sequences for efficient GOAP planning

### Strong Typing Integration

Generated agents use the existing type system:
- `IoBinding` for input/output type safety and automatic chaining
- `ZeroToOne` for cost/value metrics with validation
- `ActionQos` for quality requirements and retry policies
- `SchemaType` for data structure definitions and introspection
- `ConditionDetermination` for three-state condition logic

## Implementation Flow

### 1. User Input Analysis
```kotlin
/**
 * Analyzes user input to extract agent requirements including condition needs.
 * 
 * Identifies both explicit condition requirements (business rules, validations)
 * and implicit condition opportunities (type-based action chaining).
 * 
 * @param userInput The user's natural language requirements
 * @return Structured agent requirements with condition specifications
 */
fun analyzeUserRequirements(userInput: UserInput): AgentRequirements
```

### 2. Condition Strategy Generation
```kotlin
/**
 * Determines optimal mix of explicit and implicit conditions.
 * 
 * Analyzes requirement complexity to decide between:
 * - Explicit conditions for complex business logic
 * - Implicit conditions for simple type-based chaining
 * - Hybrid approaches for comprehensive planning
 * 
 * @param requirements Analyzed user requirements
 * @return Condition strategy with explicit/implicit balance
 */
fun generateConditionStrategy(requirements: AgentRequirements): ConditionStrategy
```

### 3. Agent Structure Generation
```kotlin
/**
 * Generates the core Agent structure with integrated condition system.
 * 
 * Creates both explicit Condition implementations and designs action
 * signatures for optimal implicit condition chaining.
 * 
 * @param requirements Structured requirements from analysis phase
 * @param strategy Condition strategy for explicit/implicit balance
 * @return Complete Agent instance with all components and conditions
 */
fun generateAgentStructure(
    requirements: AgentRequirements, 
    strategy: ConditionStrategy
): Agent
```

### 4. Code Generation
```kotlin
/**
 * Generates complete Kotlin source code with comprehensive condition integration.
 * 
 * Produces fully documented agent code with:
 * - Explicit condition implementations with proper cost assignments
 * - Type-safe action signatures for implicit condition chaining
 * - Complete KDoc documentation for all condition logic
 * - Integration with Spring Boot and Embabel frameworks
 * 
 * @param agent The Agent structure to convert to code
 * @param context Generation context with settings and preferences
 * @return Complete, compilable Kotlin source code with condition integration
 */
fun generateAgentCode(agent: Agent, context: MetaAgentContext): String
```
